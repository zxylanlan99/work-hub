## 一、接口体系概述

### 1.1 接口体系模型

本架构包含 CloudBase 云端接口 + Python 本地后端接口，交互分为三层：

```
┌──────────────────────────────────────────────────────────┐
│                    前端（浏览器）                            │
│                                                            │
│  js/auth.js  →  CloudBase Auth SDK  →  用户登录            │
│  js/db.js    →  CloudBase JS SDK   →  云数据库 CRUD        │
│  js/db.js    →  fetch()             →  Python 后端 API     │  ← v1.1 新增
│  js/ai-chat.js → app.callFunction() → 云函数 ai-proxy      │
└─────────┬────────────────────────────┬────────────────────┘
          │ HTTPS（CloudBase 托管域名）  │ HTTP（localhost:8765）
┌─────────▼────────────────┐  ┌────────▼─────────────────────┐
│    CloudBase 云端         │  │    Python 后端（本地）          │  ← v1.1 新增
│                           │  │                                │
│  云数据库 ← _openid 隔离   │  │  FastAPI → 文件解析/切片       │
│  云函数 ai-proxy           │  │  ChromaDB → 向量存储/余弦搜索  │
│  CloudBase Auth           │  │  all-MiniLM-L6-v2 → 向量化    │
└───────────────────────────┘  └────────────────────────────────┘
```

## 二、CloudBase SDK 数据操作接口（来源：12-接口设计-v1.0）

### 2.1 通用调用模式

```javascript
import cloudbase from '@cloudbase/js-sdk'

const app = cloudbase.init({
  env: 'your-env-id',
  region: 'ap-shanghai'
})

const db = app.database()

async function queryCollection(collectionName, whereClause, orderClause) {
  try {
    const { data } = await db
      .collection(collectionName)
      .where(whereClause)
      .orderBy(orderClause.field, orderClause.direction)
      .get()
    return { success: true, data }
  } catch (err) {
    return { success: false, error: err }
  }
}
```

## 三、云函数接口（来源：12-接口设计-v1.0）

### 3.1 `ai-proxy` — AI 调用代理

> **作用**：避免 AI API Key 暴露前端，统一模型路由、费用统计、错误处理。

#### 3.1.1 调用方式

```javascript
const result = await app.callFunction({
  name: 'ai-proxy',
  data: {
    messages: [{ role: 'user', content: '你好' }],
    model: 'mimo',
    stream: false
  }
})
```

#### 3.1.2 入参规范

| 参数 | 类型 | 必填 | 说明 |
|------|:----:|:----:|------|
| `messages` | `Array<{role, content}>` | ✅ | OpenAI 格式消息历史 |
| `model` | `string` | ❌ | 模型代号（`mimo`/`deepseek`/`kimi`/`doubao`） |
| `stream` | `boolean` | ❌ | 是否流式返回，默认 `false` |
| `temperature` | `number` | ❌ | 创造性（0-1），默认 0.7 |
| `max_tokens` | `number` | ❌ | 最大输出 Token，默认 1000 |

#### 3.1.3 出参规范

**成功**：
```json
{
  "content": "AI 回复的完整文本",
  "tokens": 245,
  "model": "mimo-v2.5-pro",
  "cost": 0.00073
}
```

**失败**：
```json
{
  "error": "RATE_LIMITED",
  "message": "AI 服务繁忙，请稍后重试",
  "retryAfter": 30
}
```

#### 3.1.4 模型路由规则

| 任务类型 | 路由模型 | 切换条件 |
|---------|---------|---------|
| 中文问答、知识库检索 | MiMo（主力） | 默认 |
| 代码生成、调试 | DeepSeek-V3 | 用户手动切换 / 检测到代码块 |
| 长文写作、整理 | Kimi | 用户手动切换 / 检测到长文本 |
| 简单问答、快速响应 | 豆包 Lite | 月度预算 ≥ 80% 时自动切换 |

## 三-B、知识库后端 API 接口（v1.1 新增）

> **来源**：知识库 RAG 能力扩展（文件上传 → 解析 → 切片 → 向量化 → 向量检索）  
> **技术栈**：Python FastAPI + ChromaDB + sentence-transformers (all-MiniLM-L6-v2)  
> **服务地址**：`http://localhost:8765`（配置于 `src/js/config.js` → `kbBackend.baseURL`）  
> **API 文档**：`http://localhost:8765/docs`（FastAPI 自动生成 Swagger UI）  
> **启动方式**：`cd backend && ./start.sh`

### 3-B.1 架构概览

```
前端 knowledge.html                    Python 后端 (FastAPI :8765)
      │                                       │
      │  POST /api/knowledge/upload            │  1. 保存文件到 uploads/
      │  (FormData: file + categoryId)        │  2. 返回 taskId + itemId
      │ ──────────────────────────────────►   │  3. 后台线程异步处理:
      │                                       │     parse_file() → chunk_document()
      │  GET /api/knowledge/status/{taskId}   │     → embed_texts() → store_chunks()
      │  (前端每5秒轮询)                       │
      │ ──────────────────────────────────►   │  返回 {status, progress, chunkCount}
      │                                       │
      │  POST /api/knowledge/search           │  embed_query() → ChromaDB cosine search
      │  (JSON: query, top_k, min_similarity) │
      │ ──────────────────────────────────►   │  返回 [{id, content, similarity, metadata}]
      │                                       │
      │  DELETE /api/knowledge/{itemId}       │  删文件 + 删ChromaDB向量 + 删注册记录
      │ ──────────────────────────────────►   │
```

### 3-B.3 接口详情

#### API-001：上传文件

| 项目 | 内容 |
|------|------|
| **路由** | `POST /api/knowledge/upload` |
| **Content-Type** | `multipart/form-data` |
| **用途** | 上传文件，后端后台异步处理（解析→切片→向量化→存储） |
| **支持格式** | `.pdf` `.docx` `.pptx` `.md` `.markdown` `.txt` |
| **前端方法** | `DB.uploadKnowledgeFile(file, categoryId)` |

**请求参数（FormData）：**

| 参数 | 类型 | 必填 | 说明 |
|------|:----:|:----:|------|
| `file` | `File` | ✅ | 上传的文件对象 |
| `categoryId` | `string` | ❌ | 分类 ID，默认空字符串 |
| `itemId` | `string` | ❌ | 自定义条目 ID，不传则自动生成 |

**成功响应：**

```json
{
  "success": true,
  "data": {
    "task_id": "task_a1b2c3d4e5f6",
    "item_id": "kb_x7y8z9w0a1b2",
    "file_name": "React设计模式.pdf",
    "file_size": 1048576,
    "status": "processing"
  }
}
```

**失败响应：**

```json
{
  "detail": "不支持的格式: .xlsx，支持: pdf, docx, pptx, md, markdown, txt"
}
```

**后台处理流程（异步，不阻塞响应）：**

```
1. parse_file()      — PyMuPDF/python-pptx/python-docx 提取文本  (progress: 10→30)
2. chunk_document()  — PRD 5G C-001~C-006 切片策略                (progress: 30→50)
3. embed_texts()     — all-MiniLM-L6-v2 向量化 (384维)             (progress: 50→75)
4. store_chunks()    — 存入 ChromaDB (cosine space)                (progress: 75→100)
```

| `min_similarity` | `number` | ❌ | `0.3` | 最低相似度阈值（0~1），低于此值不返回 |

## 八、优化建议（来源：接口评估分析报告）

### 8.1 建议新增接口

| 接口编号 | 接口名称 | 用途 | 原因 |
|----------|---------|------|------|
| **DB-R-xxx** | 消息列表分页 | 支持大消息量的分页加载 | 当前仅有限制条数，无分页 |
| **DB-U-xxx** | 通用状态更新 | 统一状态更新接口 | 减少重复接口 |
| **DB-D-xxx** | 通用批量删除 | 统一批量删除接口 | 减少重复接口 |

### 8.2 建议合并接口

| 建议合并接口 | 当前接口 | 合并理由 | 优先级 |
|-------------|---------|---------|:------:|
| **DB-U-xxx 批量状态更新** | DB-U-002/003（暂停/恢复目标）、DB-U-006（完成任务）等 | 多个状态更新接口逻辑相似 | 中 |
| **DB-D-xxx 批量删除** | DB-D-005（批量删除知识条目）、DB-D-006（清空回收站）等 | 批量删除操作可统一接口 | 高 |

### 8.3 建议调整的接口

| 接口编号 | 当前问题 | 建议调整 |
|----------|---------|---------|
| **AI-012** | 发送消息包含多个操作 | 拆分为发送消息和获取回复两个步骤 |
| **AGG-001** | 跨模块聚合复杂度高 | 考虑通过云函数实现 |

### 8.4 不建议合并的接口

| 接口类别 | 原因 |
|---------|------|
| **AI调用接口（AI-xxx）** | 每个AI调用有不同的action参数，逻辑差异较大 |
| **聚合查询接口（AGG-xxx）** | 每个聚合查询有不同的数据源和统计逻辑 |
| **核心CRUD接口** | 创建、读取、更新、删除操作语义明确，无需合并 |
