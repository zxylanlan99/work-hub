## 一、项目执行规则

### 1.1 基本执行流程

**规则**：先出方案，方案确认之后，有明确"执行"信号再行动。

**流程**：
1. 需求提出 → 分析 → 输出方案
2. 方案评审 → 确认/修改
3. 收到"执行"信号 → 实施
4. 完成后报告结果

### 1.2 文档变更规则

1. 所有设计文档变更需先输出变更方案
2. 变更方案确认后执行
3. 重大变更需更新相关联文档

### 1.3 代码开发规则

1. 遵循技术规范总览定义的标准
2. 先创建文件结构，再实现功能
3. 代码完成后进行自检

## 二、文件命名规范

### 2.1 文档命名

| 类型 | 格式 | 示例 |
|------|------|------|
| 需求文档 | XX-文档名-vX.X.md | 05-PRD-产品需求文档-v3.0.md |
| 设计文档 | XX-文档名-vX.X.md | 08-产品概要设计-v2.md |
| 规范文档 | XX-文档名.md | 13-技术规范总览.md |
| 规则文档 | XX-规则名.md | project-rules.md |

### 2.2 代码命名

| 类型 | 规范 | 示例 |
|------|------|------|
| HTML页面 | 小写单词 | index.html, plan.html |
| CSS文件 | 小写 | common.css, home.css |
| JS模块 | 小驼峰 | common.js, home.js |
| 工具函数 | 小驼峰 | toast.js, modal.js |

## 三、版本管理规则

### 3.1 文档版本

- 格式：v主版本.次版本
- 重大变更：主版本+1
- 小幅调整：次版本+1

### 3.2 Git提交规范

```
<类型>: <简短描述>

类型：
- docs: 文档更新
- feat: 新功能
- fix: 修复
- refactor: 重构
- test: 测试
```

## 四、会议与沟通规则

### 4.1 需求评审

- 评审前准备：输出完整方案
- 评审内容：可行性、完整性、一致性
- 评审结论：确认/修改/否决

### 4.2 设计评审

- 评审内容：UI设计、功能设计、技术方案
- 评审标准：不重不漏、一致性、可执行性

---

## 五、质量标准

### 5.1 代码质量

- 符合技术规范定义的标准
- 无重复代码
- 命名清晰可读
- 有适当的注释

### 5.2 文档质量

- 结构清晰
- 内容完整
- 无歧义
- 格式统一
