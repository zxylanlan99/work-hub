## 一、产品概述

### 1.1 一句话定义

> 不是帮你收藏更多，而是帮你**学得进去、记得住、用得上、写得出来**。

### 1.2 解决的7大痛点

| 编号 | 痛点 | 严重度 | 核心诉求 |
|------|------|--------|----------|
| P1 | 学习时间碎片化 | 高 | 只有晚上/周末有成块时间，平时零散 |
| P2 | 计划不好确定 | 高 | 目标模糊、计划常改、变更即崩盘 |
| P3 | 资讯五大难题 | 高 | 渠道乱、信息爆、不成体系、无法转化、收藏即忘 |
| P4 | AI工具混用 | 高 | 多平台混用、不知优劣、性价比不明 |
| P5 | 知识库混乱 | 高 | 学/工/生混放、过期堆积、查找困难 |
| P6 | 缺乏复习机制 | 中 | 学完即忘、知识零散、无温故计划 |
| P7 | 知识难转化 | 高 | 学了写不出、难以变实践 |

### 1.3 6大设计方向

| 方向 | 设计体现 |
|------|----------|
| 陪伴型（循序渐进） | 暖身→碎片→深度，不强制执行 |
| 多设备无缝衔接 | CloudBase Auth 云同步（登录即同步，无需 PWA 安装） |
| 收敛型（聚焦深挖） | 主题锁定模式，防止贪多 |
| 学习→实践触发器 | 知识与工作上下文自动匹配 |
| 云端统一存储 | CloudBase NoSQL 统一存储，多云即时同步 |
| All-in-One 新工具 | 一个入口使用 MiMo/DeepSeek/Kimi/豆包，智能路由+费用透明 |

> 📝 v3.0 变更：原"PWA 跨端同步"调整为 CloudBase Auth 云同步；原"分级存储（机密不出本地）"调整为云端统一存储，通过 CloudBase 安全规则实现行级数据隔离（每用户仅能访问自己 `_openid` 标记的数据）。

### 1.4 目标用户

- 主要用户：希望系统化管理个人学习的职场人士
- 使用场景：晚上/周末成块学习 + 工作间隙碎片时间
- 技术能力：具备基础互联网工具使用能力

## 二、AI 介入方案（痛点驱动）

### 痛点 1：学习时间碎片化 → 2个AI节点

#### 1C 暖身引擎

| 项目 | 内容 |
|------|------|
| 做什么 | 打开应用时推送 1-3 分钟"预热内容"，快速激活学习状态 |
| AI 输入 | 用户上次学习位置 + 当前锁定主题 + 待复习队列 + 打开时间 |
| AI 输出 | 2-3 张暖身卡片：昨日回顾（1句）+ 1道快问快答 + "上次学到哪"提示 |
| 交互规则 | 每天首次打开触发；3分钟上限；可跳过 |
| 解决什么 | 降低深度学习的心智门槛 |

#### 1D 智能续接

| 项目 | 内容 |
|------|------|
| 做什么 | 记录每次学习的精确断点，下次打开一键回到断点 |
| AI 输入 | 上次结束位置（知识条目/任务/文章） + 关联知识点 |
| AI 输出 | 上下文摘要（≤3句）+ 续接入口 + 预计剩余时间 |
| 交互规则 | 首页顶部"继续上次学习"卡片；跨天跨时段有效 |
| 解决什么 | 碎片与深度学习无缝衔接 |

### 痛点 2：学习计划难定、易变 → 5个AI节点

#### 2A 目标澄清

3-5轮对话引导模糊目标→结构化：{主题、范围、深度、时间约束、前置知识、推荐路径}

#### 2B 路径生成

生成 5-8 个里程碑 × 每个 3-6 个任务 + 推荐学习资源 + 预计总耗时 + 难度曲线

#### 2C 难度估时

基于内容量 × 用户历史速度 × 复杂度，输出 P70 置信区间的预计耗时

#### 2D 弹性排期

任务延期时自动顺延 + 标记卡住任务 + 影响分析报告。"宁可跳过，不可崩盘"。

#### 2E 进度诊断

每周自动诊断：完成率 + 卡点分析 + 调整建议 + 下周展望

### 痛点 3：资讯五大难题 → 6个AI节点

#### 3A 信源可信度

域名权威度 + 作者可信度 + 引用背书 + 历史采纳率 四维综合评分

#### 3B 内容价值

信息密度 + 原创性 + 深度（>3000字花高分）+ 实用性（含代码/案例）+ 客观性

#### 3C 知识结构对齐

零散资讯自动映射到用户知识体系分类树

#### 3D 学习素材转化

原文 → 摘要 + 知识点卡片 + 复习问题 + 输出素材

#### 3E 智能推送时机

在用户最有接收意愿的时机推送收藏内容

#### 3F 信息去重合并

识别不同来源重复内容 → 合并为综合条目 + 差异标注

### 痛点 4：AI工具混用 → 2个AI节点

#### 4A 任务分类路由

**路由决策树**：

```
用户输入 → 意图分析
  ├── 默认/通用 → MiMo（主力模型，小米大模型平台）
  │   覆盖：日常问答、概念解释、内容处理、多轮对话
  │   优势：高性价比、中文优化、响应快速
  │
  ├── 代码/编程 → DeepSeek（通过 SiliconFlow 平台）
  │   关键词：代码、报错、bug、函数、实现、框架、API
  │   语义：需要生成代码、调试代码、解释技术实现
  │
  ├── 写作/内容创作/长文 → Kimi（通过 SiliconFlow 平台）
  │   关键词：写、总结、润色、翻译、改写、大纲、长文
  │   语义：需要中文写作、内容加工、长文处理
  │
  ├── 推理/数学/逻辑 → DeepSeek（通过 SiliconFlow 平台）
  │   关键词：证明、推理、计算、逻辑、公式、推导
  │   语义：链式推理、数学计算、逻辑推导
  │
  ├── 快速响应/低成本 → 豆包（通过火山引擎平台）
  │   关键词：什么是、怎么、为什么、能否、有没有
  │   语义：简单问答、概念解释、日常极低成本咨询
  │
  └── 多轮对话 → 继承当前对话模型
       语义：是上一轮对话的延续
```

**置信度策略**：
- ≥80%：自动路由，显示"使用XXX（推荐）"
- 50-79%：显示推荐模型+理由，用户确认
- <50%：使用主力模型 MiMo，并列备选

**模型能力画像**：

| 模型 | 平台 | 强项 | 性价比 |
|------|------|------|--------|
| MiMo | 小米大模型平台 | 通用问答、中文理解、日常对话、多轮交互 | ★★★★★（主力，高性价比） |
| DeepSeek | SiliconFlow | 代码、技术问答、逻辑推理、数学 | ★★★★★（0.001元/千tokens） |
| Kimi | SiliconFlow | 中文写作、长文处理、翻译 | ★★★★（0.003元/千tokens） |
| 豆包 | 火山引擎 | 快速响应、日常问答、成本极低 | ★★★★★（Lite 0.0003元） |

> 📝 v3.0 变更：主力模型切换为 **MiMo**（小米大模型平台），移除 Claude 和 DeepSeek-R1。备选模型 DeepSeek/Kimi 统一通过 SiliconFlow 平台接入，豆包通过火山引擎平台接入。所有 AI API Key 通过 CloudBase 云函数 `ai-proxy` 代理，前端不暴露。

#### 4B 结果对比

**触发条件**：
- 用户手动勾选"对比模式"
- 路由置信度 50-79% 时 AI 建议对比
- 高价值问题自动建议对比

**对比布局**：两个模型并排展示，AI 自动标注差异

| 标注类型 | 规则 |
|----------|------|
| 核心一致 ✅ | 语义对比 >80%，核心事实一致 |
| 角度互补 🔄 | 同一主题不同角度，互不冲突 |
| 细节冲突 ⚠ | 存在事实差异（具体标注差异点） |
| 质量悬殊 📊 | 评分差距 >20分 |

**用户操作**：[采纳左侧] [采纳右侧] [合并] [继续对比] [退出]

### 痛点 5：知识库混乱 → 7个AI节点

#### 5A 强制分类引导

创建知识时 AI 自动推荐三级分类路径，用户确认

#### 5B 过期检测

技术类 >1年、非技术类 >3年自动标记"可能过时"

#### 5C 语义搜索

> 📝 v3.0 备注：MVP 阶段使用 CloudBase 正则查询实现全文检索。语义搜索（向量+BM25 双路召回）推迟至 V2.5 阶段，届时评估 CloudBase 向量数据库或外部 Embedding 服务。

#### 5D 知识体检

月度健康报告：重复内容/孤岛知识/薄弱环节

#### 5E 智能关联

自动发现知识点间隐含关联

#### 5F 知识保鲜

自动搜索最新信息更新旧知识

#### 5G 知识库人机复用设计（参考Coze知识库）

##### 设计目标

同一份知识库，AI通过切片+矢量库检索，人类通过原文阅读。同源数据，不同视图。

##### 切片策略（Chunking）

| 规则 | 内容 |
|------|------|
| C-001 | 按 Markdown H2 标题层级切分，每个切片为一个完整的 H2 段落 |
| C-002 | 目标 500-1500 字符，优先保证语义完整 |
| C-003 | 超 2000 字符 => 按自然段二次切分，Overlap 200 字 |
| C-004 | 代码块跟随所属段落 |
| C-005 | 表格保持完整，不切分 |
| C-006 | 每个切片保留元数据：来源文档ID、标题、位置、分类路径 |

**切片示例**：
```
# 微服务架构深度解析
## 一、核心概念（800字）         → Chunk 1
## 二、服务拆分策略（1500字）     → Chunk 2
### 2.1 DDD限界上下文（600字）    → Chunk 3
### 2.2 事件风暴（700字）         → Chunk 4
## 三、通信机制（2200字）         → Chunk 5（超长二次切）
  段落1-3（1100字）             → Chunk 5a
  段落4-6（1100字，重叠前200）   → Chunk 5b
```

##### 矢量库设计

| 项目 | 方案 |
|------|------|
| 嵌入模型 | text-embedding-3-small / bge-small-zh |
| 向量维度 | 512 / 1536 |
| 存储（当前） | CloudBase NoSQL `knowledge_items` 集合，MVP 阶段不索引向量 |
| 存储（V2.5+） | CloudBase 向量数据库 或 外部 Embedding 服务 |
| 检索（V2.5+） | 向量余弦 + BM25 双路召回 → RRF 融合 → Re-rank |
| Top-K | 默认 Top 5，可调 Top 10 |

##### 人类阅读视图

- 原始 Markdown 全文渲染，不受切片影响
- TOC 目录导航 + 阅读进度记录
- AI 引用标注："来源：《文档名》§章节"，可跳转

##### AI 调用流程

```
用户提问 → AI判断需查库 → 提取query → 检索 → Top 3切片入系统提示 → AI回答+来源引用
```

### 痛点 6：缺乏复习机制 → 5个AI节点

#### 6A 自适应排期（SM-2增强算法）

```
quality < 3（遗忘）: 间隔重置为1天，掌握度降级
quality = 3（勉强）: 间隔 = 当前×1.3
quality = 4（基本）: 间隔 = 当前×2.3，掌握度升一级
quality = 5（完美）: 间隔 = 当前×2.8，掌握度升一级
```

**个性化因子**：
- 用户平均衰减快于标准15% → 所有间隔×0.7
- 特定时段（如晚间）效果差 → 该时段×0.8
- 知识点复杂度高 → 增加一个额外复习节点

**7条复习原则**：

| 编号 | 原则 |
|------|------|
| RP-001 间隔递增 | 每次成功回忆后，下次间隔不短于当前间隔 |
| RP-002 失败重置 | 连续2次 quality<3 → 重置为初始间隔，标记"重点攻克" |
| RP-003 难度感知 | AI评估难度越高 → 初始间隔越短，递增越慢 |
| RP-004 实战加权 | 输出文档中使用了某知识点 → 该复习周期可跳过 |
| RP-005 上下文关联 | 当前主题中的知识点 → 间隔缩短 20% |
| RP-006 分布均匀 | 每日复习 ≤20 张，超出延后 |
| RP-007 快乐原则 | 每天前3张从"大概率能答对"的开始 |

#### 6B 智能出题

| 知识类型 | 首选题型 | 备选 |
|----------|----------|------|
| 概念/定义 | 填空题 | 选择题 |
| 过程/步骤 | 排序题 | 选择题 |
| 对比/区别 | 关联连线题 | 简答题 |
| 代码/实践 | 补全代码 | 选择题 |
| 理论/原理 | 简答题（关键词评分） | 选择题 |

**难度递进**：第1次(选择题) → 第2次(填空) → 第3次(关联) → 第4次(场景应用) → 第5次+(混合+串联)

#### 6C 串联复习

完成后AI检索关联知识点（向量>0.6）；关联知识也待复习 → 提示串联；同主题完成3个 → 提示主题串联测试

#### 6D 遗忘预测

距上次复习天数 / 遗忘曲线临界天数 = 遗忘风险比；>0.8 → 标记高遗忘风险，优先推送

#### 6E 薄弱点诊断

主题平均掌握度<2.5 → 薄弱主题；自动生成补强计划（3-5个最弱点+推荐资料）；连续3次quality<3 → "顽固项"，建议换学习策略

## 三、资讯 AI 筛选规则

### 3.1 五维评分体系

| 维度 | 权重 | 核心评估项 |
|------|------|------------|
| 信源可信度 | 20% | 域名权威度 + 作者可信度 + 引用背书 + 历史采纳率 |
| 内容价值 | 30% | 信息密度 + 原创性 + 深度 + 实用性 + 客观性 |
| 主题关联度 | 25% | 向量相似度 + 知识库关联 + 计划匹配 + 填补空白 |
| 信息新鲜度 | 15% | 发布时间 + 时效性判断 + 更新频率 |
| 可转化性 | 10% | 可摘要性 + 可操作性 |

### 3.2 分级处理

| 分数区间 | 处理方式 | 标识 |
|----------|----------|------|
| 80-100 | 强烈推荐 → AI推荐清单 | 🟢 |
| 60-79 | 推荐 → AI推荐清单 | 🔵 |
| 40-59 | 可选 → 候补清单 | ⚪ |
| <40 | 不推荐 → 不主动展示 | — |

### 3.3 去重规则

同一URL → 自动跳过；内容相似>85% → 标记"疑似重复"；60-85% → "可能相关"，展示差异摘要

### 3.4 AI标签推荐

主题标签(最多3个) + 类型标签(技术/教程/论文/博客) + 难度标签(入门/进阶/高级) + 场景标签(理论学习/实践/案例/工具)

## 四、AI推荐 → 人工确认 → 入库流程

### 4.1 完整流程

```
AI评分 ≥60 → AI推荐清单(独立Tab) → 用户浏览 → [确认入库]/[忽略]/[暂缓]
                                                │
                                                ▼
                                         入库确认弹窗
                                          |
                           ┌────────────────┼────────────────┐
                           │ 标题(AI推荐/可改) │ 分类(AI推荐/必选) │ 标签(AI推荐/可增删)
                           │ 摘要(AI生成/可改) │ 敏感级(默认公开)   │ 关联知识(AI推荐/可多选)
                           └────────────────┼────────────────┘
                                            ▼
                                    是否加入学习计划？
                                      [仅入库] [入库+加入计划]
                                            │
                                            ▼
                                   知识库正式入库
                                            │
                                ┌───────────┼───────────┐
                                ▼           ▼           ▼
                           生成复习卡片   关联学习计划   加入输出素材库
```

### 4.2 AI推荐标识规范

| 标识 | 含义 | 消失条件 |

### 8.9 资讯模块 — RSS 源管理

#### 8.9.1 功能概述

用户可管理自己的 RSS 信息源（rss_sources 集合），支持自定义源的增删改查、启用/禁用控制，以及从预设源列表快速添加。RSS 源作为资讯模块的数据输入端，配合 `rss-crawler` 云函数定时抓取文章，为 AI 筛选和推荐提供原始内容。

#### 8.9.2 RSS 源操作

| 操作 | 规则 |
|------|------|
| **新增** | 必填：源名称(≤100字)、源URL(合法 RSS/Atom Feed 地址)。系统自动验证 URL 可达性和 Feed 格式有效性；可选：分类标签、抓取频率(默认每天)。支持从预设源列表一键添加。验证失败提示错误原因，不允许保存无效源 |
| **编辑** | 可改：源名称、分类标签、抓取频率、启用/停用状态。不可改：源URL（需删除后重建）。编辑后下次抓取生效 |
| **删除** | 二次确认："此源下有 N 篇已抓取文章"，选项：[保留已入库文章(推荐)] [同时删除未入库文章] [取消]。删除后停止该源的所有定时抓取任务 |
| **启用/禁用** | 启用：立即触发一次抓取 + 恢复定时任务；禁用：停止定时抓取，已抓取文章保留。列表中显示启用/禁用状态标签 |
| **批量操作** | 支持批量启用/禁用/删除。批量删除需二次确认 |
| **查询** | 支持按名称/URL关键词搜索、按状态(启用/禁用)筛选、按分类筛选。列表展示：源名称 + URL + 状态标签 + 最后抓取时间 + 已抓取文章数 + AI平均评分 |

#### 8.9.4 爬虫升级：全文抓取

| 项目 | 规则 |
|------|------|
| **抓取策略** | 优先抓取文章全文（通过解析原文页面），而非仅使用 RSS Feed 中的标题和摘要 |
| **降级策略** | 全文抓取失败时（如页面反爬、超时），降级使用 RSS Feed 中的摘要内容，并标记 `fetchStatus: 'partial'` |
| **内容清洗** | 去除广告、导航栏、评论区、侧边栏等非正文内容，保留正文 + 代码块 + 图片链接 |
| **分页处理** | 对于分页文章，自动合并多页内容为完整正文 |
| **AI评分升级** | AI 五维评分基于文章全文进行评估，而非仅基于标题 + 摘要；全文评分结果更准确，内容价值维度（信息密度、原创性、深度）评估更可靠 |
| **存储** | 全文内容存储至 news_items 集合的 `fullContent` 字段；原始 RSS 摘要保留至 `summary` 字段作为备份 |

#### 8.9.5 资讯页面 Tab 切换风格统一

| **标签设计** | 水平标签页样式：[AI推荐] [候补清单] [全部资讯] [RSS源管理]，默认选中"AI推荐" |

```
│  │  静态托管  │  │  Auth    │  │ 云数据库  │  │ 云函数    │     │
│  │ (HTML/JS) │  │(匿名/微信)│  │ (NoSQL)  │  │(Node.js) │     │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘     │
│                    腾讯 CloudBase                             │
└─────────────────────────────────────────────────────────────┘
```

### 9.2 技术栈详情

| 层级 | 选型 | 说明 |
|------|------|------|
| 前端框架 | **无框架**，纯 HTML/CSS/ES6 原生 JS | 零代码基础可维护，8 个原型直接改造 |
| CSS | CSS Variables（`:root` 变量体系） | 白色主题统一，无 CSS 框架依赖 |
| 数据库 | CloudBase NoSQL 云数据库 | JSON 文档型，14 个集合，`_openid` 行级安全 |
| 后端 | CloudBase 云函数（Node.js 18+） | ai-proxy（AI 路由）+ data-cleanup（定时清理）+ rss-crawler（RSS 抓取） |
| 认证 | CloudBase Auth | 匿名登录为主，V2 阶段支持微信扫码登录 |
| 部署 | CloudBase 静态托管 | 1GB 存储 / 5GB 月流量（免费额度） |
| AI 主模型 | **MiMo**（小米大模型平台） | 高性价比，中文优化，兼容 OpenAI API 格式 |
| AI 备选 | DeepSeek/Kimi（SiliconFlow）+ 豆包（火山引擎） | 统一通过 ai-proxy 云函数代理 |
| 费用控制 | CloudBase 免费额度（2GB 存储、5万读/天、3万写/天、云函数 1000次/天） | 月度总成本目标 < ¥5 |

### 9.3 关键设计决策

| 决策 | 方案 | 理由 |
|------|------|------|
| 无前端框架 | 纯 HTML/CSS/JS | 零代码基础可维护，原型直接改造为产品代码 |
| CloudBase NoSQL | 14 个扁平集合 + 应用层关联 | 无需 SQL JOIN，JSON 文档天然匹配前端数据模型 |
| AI API 代理 | CloudBase 云函数 `ai-proxy` | API Key 不暴露前端，统一路由和费用控制 |
| 行级安全 | 每集合配置安全规则 `_openid == auth.openid` | 多用户数据严格隔离 |
| 匿名登录 | CloudBase Auth 匿名登录 | V1 阶段零门槛，V2 阶段升级为微信登录 |
| 白色主题 | CSS Variables 统一 light mode | 保持简洁专业，聚焦学习内容 |
| 无 PWA | 暂不实现离线模式 | 云端优先，数据实时同步优先于离线可用 |

### B. 数据库实体模型

```
CloudBase NoSQL 集合（14 个）：

业务对象1 — 学习计划体系:
  goals (目标) → milestones (里程碑) → tasks (任务)

业务对象2 — 知识存储体系:
  categories (分类) → knowledge_items (知识条目)
  knowledge_items → review_cards (复习卡片) → review_history (复习历史)

业务对象3 — 信息摄入体系:
  rss_sources (RSS源) → news_items (资讯条目)

业务对象4 — AI对话体系:
  chats (对话) → messages (消息)

业务对象5 — 知识输出体系:
  output_docs (输出文档) + scraps (碎片想法)

业务对象6 — 系统配置:
  user_settings (用户设置)

所有集合均由 CloudBase 自动注入 _id (主键) 和 _openid (用户标识)，
通过安全规则实现行级数据隔离。
应用层通过 _id 字段建立集合间关联（无 DB 级外键约束）。
```

### C. 接口统计

- **数据库操作**：通过 CloudBase JS SDK (`@cloudbase/js-sdk`) 在前端直接操作（受安全规则约束），无传统 RESTful 接口
- **云函数接口**：3 个
  - `ai-proxy` — AI 对话流式代理（SSE）
  - `data-cleanup` — 数据生命周期清理（定时触发）
  - `rss-crawler` — RSS 抓取引擎（定时触发 + 手动触发，支持全文抓取）
- **认证**：CloudBase Auth SDK（匿名登录 / 微信登录）
