API Gateway vs Service Mesh — 核心区别

两者虽然都涉及"流量管理"，但处于不同的架构层级，本质上是互补而非替代关系。

1. 架构层级不同

  • API Gateway（应用层入口）：部署在系统边界，负责 南北向流量（外部→内部）
  • Service Mesh（基础设施层）：以 Sidecar 模式注入每个服务旁，负责 东西向流量（服务间）

2. 核心关注点不同

API Gateway 关心"谁能进来、去哪"，Service Mesh 关心"服务间如何可靠通信"。

3. 选型建议

  • 如果团队刚起步、服务数量 < 10：API Gateway 就够了（Kong / APISIX）
  • 如果服务 > 20、需要灰度/熔断/可观测：引入 Service Mesh（Istio / Linkerd）
  • 两者不冲突，典型架构是"API Gateway + Service Mesh"组合使用

Istio vs Linkerd — 对比分析

您的场景（15个微服务）恰好处于决策关键期。以下是核心维度的对比：

Istio（功能全面）

  • 优点：功能丰富（流量管理、安全、可观测一体）、社区活跃（CNCF 毕业）、多协议支持
  • 缺点：复杂度高、资源消耗大（Istiod + Envoy Sidecar 约 200MB 内存/实例）
  • 适用：大型团队（>50 服务）、需要复杂流量策略

Linkerd（极简高效）

  • 优点：轻量（Rust 编写，~10MB/实例）、部署简单、性能损耗极低（<1ms P99）
  • 缺点：功能较少（仅 HTTP/gRPC）、生态较小
  • 适用：中小团队、追求简单和性能
建议：15 个服务的规模，推荐从 Linkerd 开始。等团队熟悉 Service Mesh 后，如果发现功能不够再迁移到 Istio，迁移成本可控。

Istio vs Linkerd 怎么选？

一句话总结：大团队选 Istio，小团队选 Linkerd。15 个服务规模刚好在"小"的边缘。

从三个维度来拆解：
① 复杂度：Istio 的学习曲线大概是 Linkerd 的 3 倍。如果团队没有 K8s 运维专家，选 Linkerd 更明智。
② 资源成本：Istio Sidecar 占约 200MB × 15 实例 = 3GB 额外内存。Linkerd 仅需 150MB。
③ 功能取舍：如果未来需要多集群、授权策略、Wasm 扩展 → Istio；如果只需要 mTLS + 可观测 → Linkerd 完全够用。
